ТЕХНИЧЕСКОЕ ЗАДАНИЕ
Проект: Onion-only E2EE мессенджер без хранения сообщений

1. Цель проекта

Разработать HTTP/WebSocket-приложение для приватного обмена сообщениями между двумя пользователями.

Основные требования:
- без аккаунтов;
- без хранения сообщений на сервере;
- доступ через обычный HTTP на этапе разработки;
- возможность позже добавить Tor onion service без изменения основной логики приложения;
- end-to-end encryption;
- временные комнаты;
- одноразовые приглашения;
- защита от подбора ссылок;
- минимизация метаданных.

2. Общая архитектура

Клиент:
- браузерное приложение;
- HTML/CSS/JavaScript;
- вся криптография выполняется на клиенте;
- private keys и session keys не отправляются на сервер.

Сервер:
- FastAPI;
- HTTP API для создания комнат и приглашений;
- WebSocket для realtime-обмена;
- PostgreSQL для технических метаданных;
- Redis опционально для временных очередей delay/mix;
- сообщения не сохраняются в БД.

Tor:
- на первом этапе приложение работает как обычный HTTP-сервис;
- позже добавляется Tor daemon;
- FastAPI запускается на 127.0.0.1:8000;
- Tor onion service прокидывает .onion:80 на 127.0.0.1:8000.

3. Сетевой протокол

HTTP используется для:
- загрузки frontend;
- создания комнаты;
- получения статуса комнаты;
- создания приглашения;
- уничтожения комнаты.

WebSocket используется для:
- подключения к комнате;
- обмена public keys;
- передачи encrypted packets;
- отправки control-событий;
- закрытия комнаты.

Рекомендуемые endpoints:
- GET /
- POST /api/rooms
- GET /r/{room_code}
- GET /api/rooms/{room_code}/status
- POST /api/rooms/{room_code}/destroy
- WebSocket /ws/rooms/{room_code}

4. Комнаты

Комната создаётся через POST /api/rooms.

Сервер должен:
- сгенерировать случайный room_code;
- посчитать room_code_hash;
- сохранить hash в БД;
- создать expires_at;
- создать privacy settings;
- вернуть ссылку создателю.

Room code:
- минимум 128 бит энтропии;
- URL-safe формат;
- не должен быть последовательным;
- не должен храниться в БД в открытом виде.

В БД хранится только:
- room_code_hash;
- created_at;
- expires_at;
- status;
- privacy_mode;
- max_participants.

5. Приглашения

Доступ к комнате не должен зависеть только от room_code.

Для входа используется invite_token.

Требования:
- invite_token генерируется сервером;
- invite_token одноразовый;
- invite_token имеет TTL;
- в БД хранится только invite_token_hash;
- после успешного входа invite_token помечается как used;
- повторное использование запрещено.

Рекомендуемая ссылка:
- /r/{room_code}#invite={invite_token}

Важно:
- часть после # не отправляется браузером на сервер автоматически;
- frontend сам читает invite_token из window.location.hash;
- frontend отправляет invite_token на сервер при подключении.

6. Контроль доступа к комнате

При подключении сервер должен проверить:
- room_code_hash существует;
- комната active;
- expires_at не истёк;
- invite_token_hash существует;
- invite_token не использован;
- invite_token не истёк;

После подключения второго участника:
- комната переводится в locked;
- новые подключения запрещаются;
- invite_token помечается как used.

7. Протокол выдачи и обмена ключами

Основной вариант:
- X25519 для key exchange;
- HKDF-SHA-256 для получения ключей;
- AES-256-GCM или ChaCha20-Poly1305 для шифрования сообщений.

Минимальный E2EE flow:
1. Клиент A генерирует ephemeral X25519 key pair.
2. Клиент B генерирует ephemeral X25519 key pair.
3. A отправляет public key через WebSocket.
4. B отправляет public key через WebSocket.
5. A вычисляет shared_secret через свой private key и public key B.
6. B вычисляет shared_secret через свой private key и public key A.
7. Оба клиента получают одинаковый shared_secret.
8. Из shared_secret через HKDF выводятся session keys.
9. Сообщения шифруются на клиенте.
10. Сервер пересылает только ciphertext.

Private key:
- не отправляется на сервер;
- не сохраняется в БД;
- хранится только в памяти клиента;
- удаляется при закрытии комнаты.

8. Улучшенный протокол ключей

Для высокого уровня защиты использовать Signal-style подход:

- X3DH или Noise-подобный handshake для начального обмена;
- Double Ratchet для последующих сообщений;
- новый message key для каждого сообщения;
- удаление message key после использования;
- forward secrecy;
- post-compromise security.

Минимальная версия проекта может начать с:
- X25519 + HKDF + AES-GCM.

Advanced версия:
- Double Ratchet поверх начального shared secret.

9. Проверка ключей

Нужно защититься от MITM через сервер.

После обмена public keys клиенты должны показать safety number.

Safety number:
- вычисляется из public key A + public key B + room_id;
- отображается обоим пользователям;
- пользователи сверяют его через внешний канал;
- после сверки пользователь нажимает "Verified".

Пример:
- Security code: 481-729-204

В БД можно хранить только:
- fingerprint_hash;
- verified: true/false;
- verified_at.

Нельзя хранить:
- private keys;
- session keys;
- plaintext fingerprint, если это не нужно.

10. Формат сообщений

Сервер принимает только encrypted packet.

Внешний формат WebSocket-сообщения:
{
  "packet": "base64..."
}

Внутренний формат после расшифровки на клиенте:
{
  "type": "message",
  "body": "text",
  "message_id": "...",
  "created_at": null
}

Сервер не должен получать:
- plaintext;
- sender username;
- recipient username;
- email;
- avatar;
- typing status;
- read receipts.

11. Шифрование сообщений

Каждое сообщение должно иметь:
- уникальный nonce;
- message counter;
- session id;
- ciphertext;
- authentication tag.

Требования:
- nonce не должен повторяться для одного ключа;
- сообщение проверяется через AEAD;
- повреждённые сообщения отклоняются;
- replay-сообщения отклоняются;
- старые message keys удаляются.

12. Padding

Перед шифрованием клиент добавляет padding.

Порядок:
1. plaintext;
2. add padding;
3. encrypt;
4. send ciphertext.

Размеры пакетов:
- 512 bytes;
- 1024 bytes;
- 2048 bytes;
- 4096 bytes.

Цель:
- скрыть точный размер сообщения.

Сервер должен видеть только size class, но не реальную длину plaintext.

13. Random delays

Сервер может задерживать доставку encrypted packets.

Режимы:
- off: 0 ms;
- low: 500-2000 ms;
- medium: 2000-5000 ms;
- high: 5000-15000 ms.

Задержка применяется только к encrypted packets.

Сообщения во время задержки хранятся только в RAM/Redis queue и удаляются после доставки или истечения TTL.

14. Mini-mix mode

Mini-mix mode:
- сервер собирает encrypted packets в batch;
- batch window: 3-5 секунд;
- пакеты перемешиваются;
- после этого доставляются получателям;
- порядок входа не должен совпадать с порядком выхода.

Ограничение:
- при малом количестве пользователей защита слабее;
- mini-mix не заменяет полноценный mix network.

15. Dummy traffic

Клиент может отправлять dummy packets.

Dummy packet:
- выглядит как обычный encrypted packet;
- имеет такой же размер;
- после расшифровки клиент понимает, что это пустой пакет;
- сервер не должен отличать dummy от настоящего сообщения.

Цель:
- скрыть моменты реального общения.

16. No storage

Сервер не хранит:
- plaintext messages;
- ciphertext messages после доставки;
- историю переписки;
- private keys;
- session keys;
- user identities;
- IP addresses;
- user-agent;
- cookies для идентификации.

Допустимо хранить:
- metadata комнат;
- hash приглашений;
- security settings;
- audit events без приватных данных;
- агрегированную статистику.

Запрещено логировать:
- plaintext;

- room_code;
- invite_token;

- IP;
- user-agent;

Разрешено логировать:
- internal room id;

- error code;

26. Tor integration

На первом этапе приложение работает локально:

uvicorn app.main:app --host 127.0.0.1 --port 8000

Для onion service:

HiddenServiceDir /var/lib/tor/private_messenger/
HiddenServicePort 80 127.0.0.1:8000

После подключения Tor:
- код приложения не должен меняться;
- frontend должен использовать относительные URL;
- WebSocket URL должен строиться от window.location.host.

27. Frontend URL rules

Запрещено:
- hardcoded http://127.0.0.1:8000;
- hardcoded domain;
- third-party CDN;
- external analytics;
- external fonts.

Разрешено:
- fetch("/api/rooms");
- WebSocket через window.location.host;
- локальные static files.

28. Безопасность клиента

Требования:
- private keys только в memory;
- session keys только в memory;
- очистка ключей при закрытии комнаты;
- кнопка Destroy room;
- кнопка Clear local state;
- запрет сохранения сообщений в localStorage;
- запрет сохранения ключей в localStorage;
- запрет cookies для идентификации.

29. Destroy room

При destroy:
- статус комнаты меняется на destroyed;
- WebSocket-соединения закрываются;
- RAM state удаляется;
- очереди сообщений очищаются;
- invite tokens отключаются;
- frontend очищает локальные ключи;
- frontend очищает DOM сообщений.

30. Anti-abuse

Так как IP не используется, защита от злоупотреблений строится через:
- TTL комнат;
- max active rooms;
- max participants;
- max packet size;
- max packets per connection;
- proof-of-work перед созданием комнаты;
- лимиты на WebSocket-соединение;
- автоматическая очистка idle rooms.

31. Режимы приватности

Basic:
- E2EE;
- no storage;
- room TTL;
- one-time invite.

High:
- E2EE;
- padding;
- random delays;
- key verification;
- room lock.

Extreme:
- onion-only;
- fixed-size packets;
- mini-mix;
- dummy traffic;
- no clearnet;
- no external resources.

32. Критерии готовности

Проект считается готовым, если:
- можно создать комнату;
- можно отправить invite link;
- второй пользователь может войти только по valid invite;
- после входа invite нельзя использовать повторно;
- комната блокируется после двух участников;
- клиенты обмениваются public keys;
- клиенты получают общий session key;
- сервер не видит plaintext;
- сообщения передаются через WebSocket как ciphertext;
- сообщения не сохраняются в БД;
- TTL комнаты работает;
- destroy room работает;
- padding работает;
- random delay работает;
- mini-mix mode работает;
- логи не содержат приватных данных;
- приложение работает через HTTP локально;
- приложение может быть опубликовано через Tor onion service без изменения backend-кода.
